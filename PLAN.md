# HyperSettle on Ika + Encrypt вҖ” Implementation Plan

This plan translates the HyperSettle blueprint (`/Users/rahul/Zama/HYPERSETTLE_BLUEPRINT.md`) onto the combined Ika + Encrypt stack, scoped to Solana and EVM chains.

## 1. Stack mapping

The blueprint requires three primitives. This is how each maps:

| Blueprint primitive | Stack component |
|---|---|
| Encrypted arithmetic on integers | **Encrypt** (`encrypt-anchor`) вҖ” FHE on Solana |
| Fixed-loop matching over encrypted state | `#[encrypt_fn]` DAG compiled from Rust |
| Selective decryption at settlement time | Encrypt decryption request flow |
| Cross-chain settlement without bridges | **Ika** dWallets вҖ” threshold-signed transactions on destination chains |
| Inbound cross-chain message transport | **Wormhole** вҖ” relays encrypted intent handles from EVM spokes to Solana hub |

## 4. Message schemas (the contract between components)

Three canonical payload types carried by Wormhole VAAs or Ika-signed outbound txs. Every field is either a fixed-size handle (ciphertext reference) or plaintext metadata.

**DepositIntent** (spoke вҶ’ hub, via Wormhole):
```
version:         u8
source_chain:    u16   (Wormhole chain id)
source_domain:   u32   (HyperSettle domain id)
ticker:          bytes32
asset_hash:      bytes32
epoch:           u48
encrypted_amount: bytes  (Encrypt ciphertext handle)
encrypt_proof:   bytes  (input proof / well-formedness attestation)
intent_id:       bytes32
```

**InvoiceIntent** (spoke вҶ’ hub, via Wormhole):
```
version:         u8
source_chain:    u16
source_domain:   u32
ticker:          bytes32
epoch:           u48
encrypted_amount: bytes
encrypt_proof:   bytes
recipient_chain: u16
recipient:       bytes32  (padded address)
intent_id:       bytes32
```

**SettlementOrder** (hub вҶ’ spoke, signed by Ika dWallet):
```
version:         u8
source_chain:    u16   (always Solana hub chain id)
dest_chain:      u16
dest_domain:     u32
ticker:          bytes32
asset_hash:      bytes32
plaintext_amount: u64   (already decrypted by the hub before signing)
recipient:       bytes32
intent_id:       bytes32
nonce:           u64
```

Why plaintext amount in the settlement order? Because the destination spoke must call `transfer(amount)` on a real token вҖ” and the hub has already run decryption before asking Ika to sign. This matches Mental Model 7 (Two-Phase Reveal) from the blueprint.

## 5. Phased implementation

Every phase ends with an update to `ACTIONS.md`. No phase is considered done until that update is written.

### Phase 0 вҖ” scaffold (no Ika/Encrypt code yet)
- Create all directories and placeholder `README.md` files.
- Write `docs/ARCHITECTURE.md`, `docs/CROSS_CHAIN_FLOW.md`, `docs/FHE_NETTING.md`.
- Write `packages/shared/messages.md` defining the three payload schemas.
- Top-level `README.md` summarizing the project.
- **Exit criterion:** directory structure + docs in place, nothing to compile yet.

### Phase 1 вҖ” hub state machine (plaintext first)
Per blueprint Mental Model 12 step 2 ("Implement in plaintext first"):
- `packages/hub/programs/hypersettle-hub/` Anchor skeleton.
- Instructions: `initialize`, `receive_deposit`, `receive_invoice`, `match_invoice`, `dispatch_settlement`.
- All amounts are plain `u64` at this stage. Matching algorithm runs in plaintext.
- Unit tests in `tests/` using Anchor's ts test harness with mocked Wormhole + Ika CPIs.
- **Exit criterion:** `anchor test` passes locally; netting algorithm verified against reference vectors from the blueprint.

### Phase 2 вҖ” lift amounts to Encrypt ciphertexts
- Add `encrypt-anchor` dependency.
- Define `#[encrypt_fn]` for the matching loop in `fhe.rs` (branchless, fixed-loop per blueprint Mental Model 2).
- Replace `u64` amount fields in state with Encrypt ciphertext accounts.
- Add `decrypt_settlement` instruction that requests plaintext reveal from Encrypt before dispatch.
- **Exit criterion:** hub compiles; FHE DAG typechecks; same reference vectors pass (decrypted output matches plaintext reference).

### Phase 3 вҖ” EVM spoke
- Foundry project in `packages/evm-spoke/`.
- `HyperSettleSpoke.sol`:
  - `deposit(token, amount, destDomain, recipient)` вҖ” custody SPL-equivalent ERC20, build Wormhole VAA payload with amount encrypted *client-side* (encrypted handle passed in via the transaction).
  - `invoice(ticker, encryptedAmount, proof, recipient)` вҖ” same pattern.
  - `executeSettlement(SettlementOrder, ikaSignature)` вҖ” verify threshold ECDSA signature against known Ika dWallet public key, transfer tokens.
- Ika signature verification: plain `ecrecover` against a registered dWallet address (Ika uses secp256k1 for EVM).
- Foundry tests for happy path + signature-verification failure.
- **Exit criterion:** Foundry tests green; settlement executes when Ika signature is valid, reverts otherwise.

### Phase 4 вҖ” Solana spoke
- Anchor program in `packages/solana-spoke/`.
- `deposit`, `invoice`, `execute_settlement` instructions mirroring EVM shape.
- `execute_settlement` verifies ed25519 signature from Ika dWallet.
- **Exit criterion:** `anchor test` green.

### Phase 5 вҖ” Wormhole wiring (inbound: spokes вҶ’ hub)
- Add Wormhole `IWormhole.sol` interface + encoding lib to EVM spoke.
- Use Wormhole CPI in Solana spoke to post outbound messages.
- In hub: `receive_deposit` and `receive_invoice` consume Wormhole VAAs, parse payloads, write encrypted state.
- **Exit criterion:** integration test spawns local Wormhole guardian mock, posts deposit from EVM spoke, hub state reflects it.

### Phase 6 вҖ” Ika wiring (outbound: hub вҶ’ spokes)
- Integrate Ika Solana pre-alpha program calls.
- After `decrypt_settlement` resolves, hub calls `approve_message` CPI with the `SettlementOrder` bytes.
- Ika network (or mock signer in pre-alpha) returns a signature account.
- Hub reads the signature account and emits an event that an off-chain relayer can pick up to call `executeSettlement` on the destination spoke.
- Client-side relayer in `packages/sdk/src/` that watches for settlement-ready events and submits to EVM / Solana spokes.
- **Exit criterion:** end-to-end test: deposit on EVM Arbitrum mock, invoice on EVM Base mock, settlement arrives at Base spoke, funds transferred.

### Phase 7 вҖ” SDK + demo
- `packages/sdk/` TypeScript client.
- `demo/end-to-end.ts` runs the full round trip against local mocks.
- Top-level `README.md` updated with run instructions.
- **Exit criterion:** `npm run demo` in `packages/sdk/` prints the happy-path sequence without errors.

## 6. Invariants to preserve (from the blueprint)

These must hold at every phase transition:

- **Encryption boundary:** amounts encrypted, everything else (epoch, domain, ticker, recipient, intent_id, nonce) plaintext.
- **Branchless matching:** every iteration of the matching loop runs unconditionally; results masked via `FHE.select` analog in `#[encrypt_fn]`.
- **Fixed loop bounds:** matching loop has a public `MAX_DEPOSITS` constant.
- **No encrypted storage keys:** `deposits[epoch][domain][ticker]` keys are all plaintext.
- **Two-phase settlement:** hub never holds plaintext amounts except in the `dispatch_settlement` frame that decrypts and immediately signs.
- **One hub, many spokes:** spoke programs stay small and symmetric (EVM/Solana).
